## TL;DR

Die Pipeline läuft jetzt **vollständig Excel-frei** (Default) auf
**Windows/macOS/Linux**, optional mit **Anthropic (Claude)** statt OpenAI, und
validiert den generierten Rechenkern gegen ein **festes, unabhängiges
Golden-Master-Orakel**. End-to-end auf Linux nachgewiesen; der agentische
Repair-Loop hat einen fehlerhaften Wurf selbst auf **617/617** korrigiert.

## Was sich geändert hat

1. **Excel-freies Extraktions-Backend als Default** (`--export-backend openpyxl`):
   Zellformeln + gecachte Werte + Defined Names via **openpyxl**, VBA via
   **oletools.olevba** — pure Python, kein Microsoft Excel / kein `pywin32`.
   Das frühere COM-Backend bleibt als `--export-backend com` (Windows + Excel)
   erhalten. Golden-Master-Vergleich gegen einen echten COM-Lauf:
   **0 materielle Unterschiede** (Rest dokumentierte Kosmetik/Präzision).
2. **LLM-Provider-Abstraktion:** `--provider {openai,anthropic}` + `--model`;
   einheitlicher `generate_completion`-Dispatch (OpenAI Responses- bzw.
   Anthropic Messages-API, Streaming, Truncation-Erkennung).
3. **Secret-Handling:** `*_FILE`-Pointer-Konvention — API-Key in restriktiver
   Host-Datei, `.env` enthält nur den Pointer; Key wird direkt an den
   SDK-Konstruktor übergeben (nie in einer persistenten Host-Variable).
4. **`requires-python` 3.12 → 3.11** (Code läuft nachweislich auf 3.11; ganze
   Suite grün, kein 3.12-only-Syntax).
5. **Statisches Security-Gate präzisiert** + **Laufzeit-Confinement**
   (`qa/fs_confine.py`): generierter Code darf **read-only und nur unterhalb des
   Repo-Roots** lesen; Schreiben/Netz/Subprocess/eval bleiben blockiert.
6. **Fester Golden-Master-Harness** (`qa/golden_master.py`, **Default
   `--test-mode fixed`**) statt pro Lauf LLM-generiertem Test: idiom-stabil,
   unabhängiges Orakel, **eine LLM-Stufe weniger** (~halbe Kosten). Der
   Rechenkern liefert berechnete Werte über einen stabilen Contract
   (`test_run.golden_master_outputs()`). `--test-mode llm` bleibt als Fallback.
7. **Agentische Variante fixed-tauglich:** Compare-Fehler → `repair_main`
   (Rechenkern mit den Abweichungen als Repair-Kontext neu generieren).

## Validierung (empirisch)

- **Excel-frei ≡ COM:** 0 materielle Diffs gegen `examples/Tarifrechner_KLV.xlsm`.
- **Funktionale Korrektheit:** fester Harness 5 Skalare + 612 Tabellenwerte =
  **617/617** bestanden.
- **LLM-Varianz (Kernbefund):** identischer Prompt, Korrektheit schwankt
  (566/566 → 616/617 → 150 Abweichungen). Belegt empirisch, **warum** ein
  unabhängiges Orakel nötig ist.
- **Agentische Konvergenz:** schlechter Wurf (615/617, Skalare ausgelassen) →
  ein Repair-Schritt (Abweichungen zurückgespeist) → **617/617**.
- **Tests:** 91 grün (Linux, Python 3.11 + venv); LangGraph-Graph kompiliert.

## Plattform

Default-Pfad = pure Python (openpyxl/oletools/anthropic) → Win/macOS/Linux.
COM (`--export-backend com`) bleibt optionaler Windows-Pfad (`pywin32` per
`sys_platform`-Marker; auf Nicht-Windows übersprungen). **Empirisch bisher auf
Linux gelaufen** (+ COM-Export einmal auf Windows) — macOS/Windows-Smoke steht
aus (siehe CR-003).
